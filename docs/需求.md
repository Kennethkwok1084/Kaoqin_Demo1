学生网管考勤系统需求文档
1. 引言
1.1 项目概述
本文档旨在详细描述“信息化建设处学生网管考勤系统”的功能和非功能需求。该系统旨在通过自动化方式，精确计算学生网管的月度工时，并提供数据统计与分析功能，以提升考勤管理的效率和透明度。

1.2 目标用户
学生网管： 系统主要使用者，负责处理报修任务、自主登记协助任务、查看个人工时。

管理员： 系统的日常管理者，负责数据导入、任务审核、人员管理、考勤数据导出。

超级管理员： 系统的最高权限者，负责系统全局设置，如工时规则的配置。

2. 功能需求
2.1 基础功能
登录与安全：

提供用户登录和密码找回功能。

支持用户角色权限管理（学生网管、管理员、超级管理员）。

首页 (Dashboard)：

实时展示核心数据，如报修单总数、待处理任务数、我的任务等。

提供数据仪表盘，直观展示关键指标。

显示系统运行状态，如在线用户数等。

2.2 考勤规则与工时计算
工时计算：

基础工时： * 线上任务：40分钟/单。

线下任务：100分钟/单。

爆单任务：+15分钟/单。

非默认好评：+30分钟/单。

巡检时长：按巡检机柜数量计算。

协助任务：按录入时长计算。

扣时规则：

超时响应： 报修单生成后超过24小时未处理，扣除负责区域小组每人30分钟/单。

超时处理： 从报修单生成时间起，超过48小时未完成处理，扣除实际处理人30分钟/人。

差评： 扣除负责区域小组每人60分钟/单。

上月结转： 允许将上月超出30小时的工时结转到本月，参与本月总工时计算。

自定义参数设置：

超级管理员可在“系统设置”页面自定义所有工时和扣时规则的数值。

2.3 任务管理
报修单任务管理：

数据导入： 支持批量导入A表（报修单总表）和B表（校园网维护记录）。

A表字段： 工单编号,报单企业,位置,对象,描述,预约时间,紧急程度,设备名称,合同编号,绑定设备,设备编号,是否同意开门,详细地址,品牌,型号编号,邮箱,收货地址,运维单位,处理建议,维修评价,自定义3,自定义4,自定义5,选项自定义6,选项自定义7,选项自定义8,报单时间,报单人,报单人电话,工单来源,接单时间,处理人,协助人,开始信息,对象审批,审批信息,工分申请,计划处理审批,协助审批,退回审批,关闭工单,工单状态,完工时间,工分,物料信息,总额,状态时间,单位,处理企业,报单类型,耗时,工单耗时,记账审批,记账总额,实付总额,退款信息,完成情况,品牌分类,验收信息,签字信息,评价星级,评价标签,评价内容,报单人留言,工程师留言,有话要说,故障分类

数据匹配： 系统以【报修人姓名 + 联系方式】为关键字段进行匹配，成功则自动更新任务属性。

未匹配处理： A表数据在B表中无法匹配时，默认标记为“线上单”。无法匹配的数据在后台标记为"未匹配记录"供管理员查看。

CRUD功能： 支持对报修单的增、删、改、查，以及批量操作。

筛选功能： 支持按各种字段（如时间、处理人、地点等）进行筛选和组合筛选。

学生网管操作：

线下单标记： 学生网管可自主将本月报修单标记为线下单，并需填写“检修结果”、“检修内容”和“检修图片”。

手动填写线下单： 学生网管可手动录入线下任务，填写相关信息，系统将等待与A表数据匹配。

重复处理机制： 当学生网管手动填写线下单，并与A表数据匹配后，系统需要一个防止重复计算的机制，例如当A表某条数据已通过学生网管手动填写并匹配，则不再重复计算线上工时。

协助任务管理：

数据来源： 支持管理员导入协助任务表，以及学生网管自主登记。

协助任务表字段： 编号,协助日期,协助地点,协助事项,协助任务时长,协助时长-自定义时长（单位分钟）,补充内容,补充协助时间（24小时制）,提交人,提交时间

自主登记： 学生网管可填写“协助日期”、“协助地点”、“协助事项”、“协助时间段”或“协助时长”，系统自动进行时间换算。

审核流程： 学生网管登记的协助任务需经管理员或超级管理员审核通过后，方可计入工时。

2.4 人员管理
人员信息管理：

支持对学生网管人员的增、删、改、查。

支持设置人员的“在职”和“离职”状态。

离职逻辑： 离职人员的账号将被禁用，但其历史考勤数据将永久保留。

支持分组功能，管理员可为人员进行分组，用于考勤扣时规则的计算。

2.5 考勤数据管理与导出
数据查看：

管理员和超级管理员可查看所有人员的考勤数据。

学生网管只能查看自己的考勤数据。

数据导出：

支持将考勤数据导出为Excel文件。

Excel结构：

工作表1 (汇总): 序号,班级,姓名,上月结转时长,报修单协助时长,日常监控协助时长,协助任务协助时长,工作总时长,签名。

工作表2 (报修单工作时长-x月): 序号,班级,姓名,报修单总数,响应超时单数,处理超时单数,超时扣除时间数,好评单数,好评折算时间数,汇总时间数,签名。

工作表3 (协助任务-x月): 序号,班级,姓名,协助任务名称,协助任务地点,协助任务日期,协助任务时间段,协助任务小时数,签名。

工作表4 (导出信息): 包含所有可自定义的参数信息，如好评加多少、超时扣多少等。

电子签名：

未来规划支持电子签名功能。

2.6 报表统计与展示
报修单趋势：

展示每月报修单总数、好评单数等趋势图表。

区域问题统计：

通过图表展示不同楼栋、宿舍的报修单数量，快速定位问题高发区域。

问题类型词云：

将报修单内容进行关键词提取，以词云形式展示主要问题类型。

个人工时排名：

展示学生网管的工时贡献排名，并可查看个人的工时构成。

3. 非功能需求
3.1 技术选型
后端：

语言： Python 3.12+ / 3.13+（推荐使用pyenv管理版本）

数据库： PostgreSQL 15+

缓存： Redis 7+（可选）

版本控制： Git

前端：

框架： Vue 3

UI组件库： Ant Design Vue

状态管理： Pinia

路由： Vue Router

HTTP客户端： Axios

数据可视化： ECharts

构建工具： Vite

3.2 性能与稳定性
系统需具备高可用性和稳定性，确保数据导入、计算和查询的流畅性。

在批量导入大量数据时，系统响应时间需控制在合理范围内。

3.3 用户体验
界面设计需简洁、直观，符合学生网管和管理员的操作习惯。

提供清晰的提示信息和操作反馈，避免用户困惑。

3.4 安全性
保障用户登录信息的安全。

对不同角色设置严格的权限控制，防止越权操作。

保护导入和导出的数据，防止泄露。

3.5 可维护性与可扩展性
系统架构应具备良好的可扩展性，以便未来添加新功能，如大模型辅助识别、电子签名等。

代码结构清晰，易于维护。

3.6 测试与质量保障
后端测试：

单元测试： 对后端每个函数和模块进行单元测试，确保其功能正确性。

集成测试： 验证后端服务与数据库、缓存等组件的集成是否正常。

API测试： 确保所有API接口的功能、性能和安全性符合预期。

前端测试：

组件测试： 确保每个UI组件在不同状态和输入下的行为正确。

状态管理测试： 验证 Pinia 状态管理逻辑的正确性。

端到端（E2E）测试：

模拟用户从登录到完成报修单处理、查看报表等完整业务流程，确保整个系统的功能和流程无误。
