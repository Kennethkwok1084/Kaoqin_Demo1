# 修改方案审核稿（2026-02-25）

## 1. 文档目的
本稿用于审核“先改什么、为什么改、改到什么程度算完成”。
当前阶段仅给出修改方案，不执行代码改动。

## 2. 评审范围
- 工时/考勤计算算法修复（后端）

## 3. 现状问题确认（已定位）

### 3.1 月度统计时间边界问题（P0）
- 现象：按月查询使用“`<= 月末日期`”时，月末当天大部分带时分秒的数据可能被漏算。
- 影响：月报、个人总工时、排名可能偏低。
- 涉及位置：
  - `backend/app/services/work_hours_service.py`

### 3.2 超时响应判定问题（P0）
- 现象：当任务有 `response_time` 时，现有逻辑可能直接判定“不超时”，未比较“响应耗时是否超过24小时”。
- 影响：超时响应罚时漏算。
- 涉及位置：
  - `backend/app/services/work_hours_service.py`
  - `backend/app/models/task.py`

### 3.3 计算口径分叉（P0）
- 现象：存在“服务层计算”与“模型层计算”两套路径，批量重算和月汇总可能走不同算法。
- 影响：同一任务在不同接口出现不一致结果。
- 涉及位置：
  - `backend/app/services/work_hours_service.py`
  - `backend/app/services/work_hour_automation.py`
  - `backend/app/models/task.py`
  - `backend/app/api/v1/repair.py`（存在 API 路径直接触发 `task.update_work_minutes()`）

### 3.4 爆单规则口径冲突（P0，需产品确认）
- 发现两种互斥口径：
  1) 爆单固定 15 分钟（不叠加基础工时）
  2) 爆单在基础工时上 +15 分钟
- 影响：历史数据重算结果会明显变化。
- 涉及位置：
  - 业务脚本与测试口径存在差异（需统一）

## 4. 后端修改清单（建议）

## 4.1 P0（必须先做）

### A. 修复月度查询时间边界
- 修改点：
  - 月度统计按“`[月初, 次月月初)`”查询，避免 `<= 月末 00:00:00` 的漏算。
- 验收标准：
  - 月末 23:59:59 的任务计入当月。
  - 新增边界测试通过。

### B. 修复超时响应判定
- 修改点：
  - 计算超时时，不以“是否有 response_time”直接返回；应比较 `response_time - report_time`。
  - 未响应场景仍按“当前时间 - report_time”判定。
- 验收标准：
  - 已响应但超过24小时的任务仍被判定超时响应。
  - 相关单测与集成测试通过。

### C. 收敛为单一计算入口（Single Source of Truth）
- 修改点：
  - 定义“单一入口”：所有工时计算/重算/汇总/API 写路径必须经过唯一核心函数（建议统一到 `WorkHoursCalculationService.calculate_task_work_minutes`，或其唯一包装函数）。
  - 禁止业务路径直接调用 `RepairTask.update_work_minutes()` 作为最终口径来源，仅允许在核心函数内部受控调用。
  - 批量重算、自动化任务、月度汇总、API 接口写路径均调用同一套核心函数。
  - 补全路径清单（实施必须逐项收敛）：
    - `backend/app/services/work_hours_service.py`
    - `backend/app/services/work_hour_automation.py`
    - `backend/app/api/v1/repair.py`
    - `backend/app/models/task.py`（保留为底层能力，不作为口径源）
- 验收标准：
  - 同一任务在“详情、重算、月汇总、API直接修改后查询”四处结果一致。
  - 全仓检索不存在新的业务路径绕过单一入口直接落库工时。

### D. 统一爆单口径并冻结规则版本
- 修改点：
  - 明确爆单业务规则（固定15 或 基础+15 二选一）。
  - 增加“当前生效口径”声明，作为实施期间唯一判定标准，避免重复争议。
  - 在配置/常量中固化规则版本并记录生效日期。
  - 在文档中一次性写明历史追溯策略（全量追溯 / 指定月份追溯 / 不追溯）。
- 验收标准：
  - 文档、代码、测试三者口径一致。
  - 历史数据重算策略明确（是否追溯）。

#### D-1. 当前生效口径声明（审核模板，需审批后生效）
- 当前生效口径：`爆单固定15分钟（不叠加基础工时），可与惩罚叠加，最小值为0`。
- 生效日期：`YYYY-MM-DD`（审批通过后填写）。
- 历史追溯策略（三选一，审批时勾选）：
  - [ ] 全量追溯（历史全部重算）
  - [ ] 指定月份追溯（起始：`YYYY-MM`）
  - [ ] 不追溯（仅生效日期后新数据）

### E. 时间与时区最小修复（P0）
- 修改点：
  - 所有比较时间统一为 UTC-aware `datetime`；禁止 naive/aware 混算。
  - 关键路径先做最小改造：工时计算、超时判定、月度边界查询、导入时间解析。
  - 对外统一约定：存储 UTC，展示层再做本地化。
- 验收标准：
  - 运行中不再出现 naive/aware 混算异常。
  - 月末边界与超时判定在 UTC 下可重复、可回归。

## 5. Vue → Flutter 迁移方案（渐进式，不停机）

## 5.1 迁移原则
- 后端 API 不重写，Flutter 复用现有接口。
- 先迁高价值页面，旧 Vue 保持可用。
- 每阶段可独立发布，避免大爆炸式重构。

## 5.2 迁移阶段

### 阶段 0：准备（1 周）
- 输出接口清单与字段契约（OpenAPI + 实际返回体抽样）。
- 确认认证机制（Access/Refresh）与错误码映射。
- 产出 Flutter 项目骨架与目录规范。

交付物：
- API 契约文档
- Flutter 基础工程（网络层、鉴权、路由、状态管理）

### 阶段 1：核心流程（2 周）
- 登录/登出/拉取用户信息
- 仪表板
- 任务列表（只读优先）

交付物：
- 可用的 Flutter MVP（核心可浏览）

### 阶段 2：业务主流程（2-3 周）
- 任务处理、工时查看、统计页
- 常用筛选与分页

交付物：
- 可替代 Vue 日常使用的 Flutter 版本

### 阶段 3：管理与导入（2 周）
- 成员管理
- 数据导入（如移动端不适合可保留在 Web 管理端）
- 设置与系统管理

交付物：
- 双端并存策略：Flutter（业务）+ Web（管理）

## 5.3 技术选型建议（Flutter）
- 状态管理：Riverpod（或 Bloc，二选一，建议 Riverpod）
- 网络：Dio + 统一拦截器
- 路由：go_router
- 序列化：json_serializable
- 本地存储：flutter_secure_storage（token）+ shared_preferences（轻量配置）

## 5.4 风险与规避
- 风险：接口返回格式不一致导致客户端适配成本高。
  - 规避：先做 API 契约治理，再迁移页面。
- 风险：算法口径未统一，Flutter 展示结果与后台报表不一致。
  - 规避：先完成 P0 算法修复后再大规模迁移。
- 风险：导入和复杂管理页面在移动端体验差。
  - 规避：保留 Web 管理后台，Flutter 重点覆盖高频业务流。

3. 后端 P1 + Flutter 阶段1并行
