# Flutter 端架构规划与开发阶段

## 1. 核心原则：明确前后端职责边界
为了避免重走 Vue 前端“偷偷兜底、补口径、做转换”的老路，我们在 Flutter 端严格执行贯穿始终的**“前端只做前端事”**原则：

### 前端（Flutter）只负责：
- 调用接口、路由跳转
- 表单数据收集、校验与提交
- 列表展示、详情展示
- 各种 UI 状态的精确呈现：加载态、空数据态、错误态
- 基于权限字段的 UI 展示控制（显示/隐藏入口）
- 各种本地设备能力如：文件选择与上传下载

### 后端（FastAPI）严格负责：
- 核心业务计算：工时计算、统计汇总、排行榜规则
- 业务流转：任务状态的转换逻辑与约束限制
- 数据处理：导入导出解析、数据结构抹平、默认值补齐
- 后端应确保返回的 DTO/JSON 格式稳定，不需要前端再去二次映射“缺省值”或“猜测类型”。

**结论**：在 Flutter 的 UI 层和 API 库中间，仅保留极其轻量的 DTO 解析转换即可，将业务计算完全推给后端。

## 2. 核心技术栈收紧
- **网络请求**：全局只使用 `Dio` + `Retrofit`。
- **Repository层**：只做“请求转发 + 模型反序列化映射”，拒绝编写任何业务判断逻辑分支。
- **状态管理**：推荐使用 Riverpod 或 Bloc（视团队现状而定，以响应式和状态分离为基准）。

## 3. 基础目录骨架 (Feature-First)
```text
lib/
├── core/         # 核心基础设施 (路由, 网络客户端, 主题, 常量配置等)
├── features/     # 按业务模块聚合 (login, dashboard, tasks, work_hours 等)
└── shared/       # 公共组件 (通用的 UI Widget, Utils, 基础模型)
```

## 4. 模块对齐清单 (参照 Vue 现状)
Flutter 将完全对齐现有 Vue 前端的业务模块：
- **登录 / Auth**
- **仪表板 / Dashboard**
- **任务管理 / Tasks**
- **工时考核 / Work Hours / Attendance**
- **成员管理 / Members**
- **统计报表 / Reports**
- **系统设置 / Settings**

## 5. 开发阶段执行顺序

### 第一阶段：最小骨架与核心主流程
- **目标**：跑通鉴权拦截和主导航流程。
- **内容**：搭建 `core` + `shared` 基础件（`main.dart`、全局路由、导航壳、响应式布局体系、认证状态拦截），并实现 **登录**、**应用壳 (App Shell)**、**仪表板**、**任务列表**、**工时页** 的 UI 占位与页面流转。

### 第二阶段：次级核心模块与边缘辅助模块
- **目标**：补齐所有业务板块的静态结构。
- **内容**：搭建 **成员管理**、**统计报表**、**系统设置** 的框架，完成对应路由绑定。 

### 第三阶段：业务对齐与真实联调
- **目标**：打通前后端血脉，全量替换 Mock。
- **内容**：把前面占位的数据全面换成真实 `Retrofit` 接口；全面补齐并测试异常处理（全局错误拦截、Token 失效、无权限页面、列表空态、骨架屏加载态）。
