# 任务管理系统重构报告

## 1. 需求分析

### 1.1 核心需求
- **数据导入完整性**：支持A表所有字段的完整导入，不仅限于当前的部分字段
- **智能任务分类**：通过A/B表匹配，根据B表"检修形式"字段准确判断线上/线下任务
- **状态自动映射**：根据A表"工单状态"字段自动设置任务的最终状态
- **精确工时计算**：实现基于任务类型和标签的差异化工时计算逻辑
- **爆单任务管理**：支持管理员手动标记爆单任务，并实现独立的工时计算
- **监控任务登记**：支持手动登记日常监控任务的时长

### 1.2 业务场景
- **月度考勤统计**：为HR部门提供准确的员工工时数据
- **绩效评估支持**：通过任务完成情况和工时数据评估员工表现
- **资源配置优化**：基于线上/线下任务分布优化人员配置
- **异常处理追踪**：识别和处理超时响应、超时处理、差评等异常情况

## 2. 系统要求

### 2.1 功能要求

#### 数据导入模块
- **A/B表智能匹配**：基于员工信息实现精确匹配
- **全字段导入**：保存A表所有原始字段数据
- **状态映射**：工单状态 → 任务状态的自动转换
- **默认处理**：A表未匹配记录默认设为线上任务

#### 任务分类系统
- **线上/线下判断**：基于B表"检修形式"字段
- **任务类别识别**：网络、设备、软件维修的自动分类
- **标签管理**：爆单、非默认好评、异常扣时标签

#### 工时计算引擎
- **差异化计算**：爆单任务独立计算，非爆单任务叠加计算
- **实时更新**：标签变更时自动重新计算工时
- **异常处理**：支持负工时保护（最小值为0）

#### 管理功能
- **批量操作**：支持批量标记爆单任务
- **数据审核**：导入数据的验证和错误处理
- **权限控制**：不同角色的功能访问权限

### 2.2 技术要求

#### 性能要求
- **导入效率**：支持大批量数据导入（>1000条记录）
- **计算性能**：工时计算响应时间 < 100ms
- **并发支持**：支持多用户同时操作

#### 数据完整性
- **事务保证**：导入过程的原子性操作
- **数据验证**：字段格式和业务规则验证
- **错误恢复**：导入失败时的回滚机制

#### 扩展性
- **模块化设计**：各功能模块独立可扩展
- **配置化**：工时标准、状态映射等支持配置调整
- **API友好**：提供完整的RESTful API接口

## 3. 具体参数标准

### 3.1 工时计算标准

#### 基础工时
| 任务类型 | 基础工时 | 说明 |
| -------- | -------- | ---- |
| 线上任务 | 40分钟/单 | 远程处理的维修任务 |
| 线下任务 | 100分钟/单 | 现场处理的维修任务 |
| 爆单任务 | 15分钟/单 | 爆单情况，独立计算工时，不与其他标签叠加，但与异常扣时叠加 |
#### 附加标签工时
| 标签类型 | 工时调整 | 适用条件 |
| -------- | -------- | -------- |

| 非默认好评 | +30分钟/单 | 可与基础工时叠加 |

#### 异常扣时标准
| 异常类型 | 扣时标准 | 判断条件 |
| -------- | -------- | -------- |
| 超时响应 | -30分钟 | 响应时间超过规定标准 |
| 超时处理 | -30分钟 | 处理时间超过规定标准 |
| 差评 | -60分钟 | 用户评价为差评 |

### 3.2 工时计算公式

#### 爆单任务
```
最终工时 = 15分钟
```

#### 非爆单任务
```
最终工时 = max(0, 基础工时 + 附加标签工时 - 异常扣时)
```

### 3.3 状态映射规则

#### A表工单状态 → 系统任务状态
| A表状态 | 系统状态 | 说明 |
| ------- | -------- | ---- |
| 已完成 | COMPLETED | 任务已完成 |
| 进行中 | IN_PROGRESS | 任务处理中 |
| 待处理 | PENDING | 等待处理 |
| 已取消 | CANCELLED | 任务已取消 |
建议在系统设置中可以自定义这些对应的字段，比如已进行所对应的COMPLETE，我可以在系统设置中修改对应COMPLETE所对应的字段。
#### B表检修形式 → 任务类型
| B表检修形式 | 任务类型 | 基础工时 |
| ----------- | -------- | -------- |
| 远程 | 线上任务 | 40分钟 |
| 现场 | 线下任务 | 100分钟 |
| 未匹配 | 线上任务 | 40分钟（默认） |
建议在系统系统设置中可以自定义这些对应的字段，比如线上任务、线下任务我可以自定义字段来对应他们。

## 5. 重构方案

### 5.1 现状分析

#### 当前系统问题
1. **数据导入不完整**：只导入部分A表字段，缺失重要业务数据
2. **任务分类逻辑错误**：基于维修类型而非B表检修形式判断
3. **工时计算不准确**：爆单任务按标签叠加而非独立计算
4. **状态映射缺失**：未根据A表工单状态设置任务状态
5. **管理功能不足**：缺少批量操作和爆单标记功能

#### 技术债务
- 数据模型字段不足
- 业务逻辑分散在多个模块
- 缺少统一的工时计算服务
- API接口不完整

### 5.2 重构策略

#### 渐进式重构原则
- **向后兼容**：确保现有功能不受影响
- **分阶段实施**：降低重构风险
- **充分测试**：每个阶段都进行完整测试
- **数据安全**：重构过程中保证数据完整性

#### 重构优先级
1. **核心逻辑重构**（高优先级）
2. **数据模型扩展**（高优先级）
3. **功能增强**（中优先级）
4. **界面优化**（低优先级）
